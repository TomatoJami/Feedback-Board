# Предложение по архитектуре SaaS

Чтобы превратить эту доску отзывов в платформу SaaS (Software as a Service), нам нужно внедрить многоарендность (multi-tenancy). Это позволит одному развернутому приложению обслуживать несколько независимых клиентов (арендаторов).

## 1. Стратегия многоарендности: «Панель управления»
Основная идея заключается в переходе от приложения для одного проекта к платформе для множества проектов.

### Новая коллекция: `workspaces` (или `projects`)
Каждый пользователь начинает с рабочего пространства по умолчанию.
- **Поля**: `name`, `slug` (уникальный), `owner` (связь с пользователями), `plan` (free/pro), `settings` (JSON для темы, логотипа и т. д.).
- **Доступ**: Только владелец (и приглашенные участники команды) могут просматривать/редактировать настройки рабочего пространства.

### Масштабирование коллекций
Каждая существующая коллекция данных должна быть привязана к рабочему пространству:
- `suggestions` → добавить поле `workspace` (relation)
- `categories` → добавить поле `workspace` (relation)
- `comments` → (уже связаны с предложениями, поэтому наследуют привязку)

### Структура маршрутов (Routing)
- **Публичный доступ**: `your-saas.com/w/[workspace-slug]`
- **Админка**: `your-saas.com/dashboard` (где пользователи управляют своими рабочими пространствами)

## 2. Тарифные планы и лимиты

### Бесплатный тариф (Free)
- **Лимит**: 1 рабочее пространство.
- **Приватность**: Только публичные доски.
- **Брендинг**: Футер "Работает на YourSaaS".
- **Участники**: Только владелец.

### Профессиональный тариф (Pro) — $49/мес
- **Лимит**: 10 рабочих пространств.
- **Приватность**: Приватные доски (доступ через SSO или пароль).
- **Кастомизация**: Собственный домен (например, `feedback.customer.com`).
- **Команда**: Возможность пригласить до 5 модераторов.
- **Дополнительно**: Приоритетная поддержка, кастомные эмодзи, внутренние комментарии (только для команды).

## 3. Ключевые технические вызовы

1. **Собственные домены**: Потребуется обратный прокси (например, Caddy или Nginx), который умеет динамически выпускать SSL-сертификаты для доменов клиентов.
2. **Изоляция данных**: Правила API в PocketBase ДОЛЖНЫ включать проверку `workspace.owner = @request.auth.id` для всех действий по управлению, чтобы предотвратить утечку данных между клиентами.
3. **Приглашения в команду**: Потребуется коллекция `workspace_members` для управления правами доступа участников.

## 4. Первые шаги для создания MVP
1. **Обновление схемы**: Добавить коллекцию `workspaces` и связь с ней в `suggestions`.
2. **Обновление Middleware**: Определять слаг (slug) рабочего пространства из URL и внедрять фильтр по workspace во все запросы к API.
3. **Создание рабочих пространств**: Простой интерфейс в `/settings` или `/dashboard` для именования и создания первого рабочего пространства.
